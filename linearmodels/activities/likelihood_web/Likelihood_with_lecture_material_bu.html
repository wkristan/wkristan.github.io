In a couple of weeks we will learn to use the differences in likelihoods of models as the basis for drawing scientific conclusions, using the method of support. To prepare you to use this new approach we will spend this week working with likelihoods to better understand how they are calculated, and what they tell us about our models. The examples we use today make use of things you already know how to do in R, like calculating a mean and fitting a regression line to some data, but we will learn now how to interpret those familiar skills in the context of maximum likelihood estimation.

Maximum likelihood estimation is, as the name implies, an estimation method. The estimation method that the statistics we've been using all semester are based on has been least squares - that is, estimates of group means and best-fit lines have been the ones that minimize the squared residuals around the estimates. Maximum likelihood is similar, in that it too is a way of finding estimates that are the best possible by some criterion, but the criterion is that maximum likelihood estimates are those that have the highest likelihood given the data. What, then, is a likelihood?

Likelihood functions and probability distributions

A good place to start understanding likelihood is through probability distributions. If you recall, a probability distribution gives all the possible values of a variable and their probabilities of occurring.

Likelihood functions are derived from probability distributions, but they are different in a critical way:

  • Probability distributions treat the population parameters of the distribution as known, and calculate the probability of particular outcomes given those known parameters.
  • Likelihood functions treat the data as known values, and calculate the likelihood of the unknown population parameters we are estimating given the known data.

To understand the difference between them, and the various ways we will see likelihood used, it's useful to refresh memory about what probability distributions are and how they work. Let's start with a discrete probability distribution, the Poisson (we will see the Poisson distribution again when we learn about loglinear models, so this is a preview of coming attractions).

Discrete probability distributions give the probabilities of outcomes that can only take particular values - the Poisson is used to calculate the probability of counts, which can only be nonnegative integer numbers (0, 1, 2, 3, etc.).

For example, if we have data on numbers of white blood cells (WBC) on slides prepared from blood smears (i.e. microscope slides with a small amount of blood) the data will be in the form of discrete counts. If the WBC on a slide are independent of one another (and if the patients are independent as well) then we can expect the probability of getting x WBC on a slide given a mean count of λ would be:

P ( x ∣ λ ) = e − λ λ x x !

A Poisson distribution has only one parameter, λ, which is both the mean count and the variance in counts (they are the same in Poisson-distributed count data), and we need to know its value to calculate the probability of a WBC count - how do we know λ? If we have a sample of data we can calculate λ by taking the mean of the counts - across 100 blood smears we might find that the mean number of blood smears λ = 2.3. We would then have values for all the terms in the equation, and can calculate the probability of getting x = 0 WBC on a slide given that λ is equal to 2.3 - in other words, P(x = 0 | λ = 2.3) is:

P ( x ∣ λ ) = e − λ λ x x ! = e − 2.3 2.3 0 0 ! = 0.100 ( 1 ) 1 = 0.100

The probability notation, P(x = 0| λ = 2.3) says "the probability that x = 0 given that λ = 2.3", where the vertical line means "given" - anything to the right of the line is treated as a known value. Given a mean number of WBC of 2.3 the probability of getting a slide with 0 WBC is 0.100 - or, equivalently, we can expect 10% of smears to have no WBC.

Since this is a discrete probability distribution we can get the probability of any of the possible outcomes (x = 1, x = 2, etc.), but we have to do this calculation for each one - we could calculate the probabilities of counts from  0 to 10 by plugging each in for x, and that would give us:

Counts can go up to positive infinity, but by the time we've calculated the probability of 10 WBC for this Poisson distribution the sum of the probabilities is very close to 1 - if we lump the rest of the possible counts into an "over 10" group the sum of the probabilities of every number from 11 to ∞ still collectively has a probability less than 1x10-16.

The fact that the sum of the probabilities is equal to 1 is an important part of the definition of a probability distribution - probability distributions give probabilities for every possible outcome for a variable, and when you look at a smear the probability of one of the possible counts occurring is equal to 1.

We can calculate the probabilities of outcomes that are discrete like this and interpret them as the chance that something will happen if we conduct a single trial - in this case a trial would be drawing a blood sample, making a smear, and counting WBC on it. A randomly selected smear has a probability of 0.1003, or a 10.03% chance, of having no WBC, a 23.06% chance of having one WBC, and so on.

To use the Poisson distribution as a likelihood function we just have to... change nothing at all about the formula. The only thing we have to do is to fundamentally change how we think about it.

Likelihoods treat the data values as known and calculate the likelihood of unknown parameters given the known data. To "convert" the Poisson probability distribution into a Poisson likelihood function we just need to change the notation on the left side of the equation:

L ( λ ∣ x ) = e − λ λ x x !

This is seemingly a minor change since the calculation we do will be exactly the same, but it fundamentally changes what the function is doing. In a likelihood function (L) the term that is unknown is the parameter that we wish to estimate (λ), and it's the data that are considered known values. This should make sense to you, because... we know what the data are, right? It's the only thing we do know - the parameter has to be estimated from data, and we almost never know what its value really is.

We can use likelihood to estimate λ, though. To do this, we can try out different values for λ and see which one has the highest likelihood given the data - that is, the maximum likelihood estimate is the value for λ that has the highest likelihood given the data.

If we have just a single blood smear with no WBC in it, the value of λ with the highest likelihood is the data value - if we plug in 0 for λ with an observed data value of x = 0 we get:

L ( λ ∣ x ) = e 0 0 0 0 ! = 1

Since the largest value the probability distribution we're using as our likelihood function is capable of producing is a 1 we can be certain that λ = 0 is the maximum likelihood estimate.

When we have more than one data value we can combine the likelihoods by multiplying them - if we had one smear with a count of 0 and another with a count of 3 we could calculate the likelihood of an estimated value of λ for each and multiply them together to get the likelihood of the estimate of λ given the whole (small) data set:


A couple of things to notice:

  • The likelihood function using only X = 0 is the blue line, and it maximizes at the data value of 0
  • The likelihood function using only X = 3 is the orange line, and it maximizes at its data value of 3
  • The yellow line is the likelihood function for both of the data values, and it is calculated by multiplying the two likelihood functions together - that is, a value for lambda is selected and its likelihood for x = 0 and for x = 3 are both calculated, and then their likelihoods are multiplied together to get the likelihood given both of the data values. The yellow line maximizes at 1.5, which is also the average of 0 and 3 - this means that the arithmetic mean of the data is also the maximum likelihood estimate for lambda
  • The probability distribution is discrete, but the likelihood function is continuous - the mean of a set of counts can be a decimal number, so the mean isn't constrained to be a discrete integer number the way that counts are. Consequently, the Poisson likelihood function for lambda is a continuous function, even though the probability distribution is discrete

Another important difference between probabilities and likelihoods is that probabilities have a natural, meaningful interpretation as single values, but likelihoods do not. For example, we saw that a probability of 0.1 for x = 0 when λ = 2.3 told us that we can expect 10% of the smears to have 0 blood cells. But, the maximum likelihood on the graph for the whole data set of 0.028 doesn't mean anything in particular - it is meaningful that it's the maximum likelihood, but whether the maximum of the likelihood function is a large number or a small number doesn't matter at all. Likelihoods are only interpretable in a relative sense, in comparison with other likelihoods, but a single likelihood doesn't tell us anything useful.

Let's move on to an example with a continuous variable now - the basic idea is exactly the same, but since continuous data are described by continuous probability distributions, such as the normal distribution, we need to use a normal likelihood function to describe them.

Using the normal distribution as a likelihood function

The normal distribution has been our constant companion (if perhaps not our best friend) all semester long - we know that data are often normally distributed, and we know that GLM assumes normally distributed residuals.

The normal distribution is a continuous probability distribution, and its probability density function has the equation:

p ( x ∣ μ , σ ) = 1 2 π σ 2 e − 1 2 [ x i − μ σ ] 2

To use this equation we need to know the mean (μ) and the standard deviation (σ), and then we can plug in any data value (xi) to do the calculation - the other terms in the equation are constants (e, π, 1, 2) with known values. What we get, p(x |μ, σ). is a probability density for x given known values for μ and σ rather than a probability. Since a continuous variable can take any value on the number line to an infinite level of precision any single value is just a point, with no length or width. For a continuous variable probability is defined as an area under the curve - for example, here's the normal curve for values of x with a mean of μ = 10 and σ = 0.1:

If you look at the y-axis values you'll see that these can't possibly be probabilities, because they go above 1, and probabilities have to fall between 0 and 1. The area under the curve can still be 1 even though the height of the curve exceeds 1, because areas are bases multiplied by heights - normal distributions extend from -∞ to ∞ but the height of the curve approaches 0 as you move away from the mean in the middle of the curve, so most of the height of the curve falls in the range that's graphed here. As such, the bulk of the area falls in a narrow range between 9.7 and 10.3, and with a narrow base the area can still be 1 when the height is greater than 1 (right? For example a rectangle with a base of 2 and a height of 0.5 is 2 x 0.5 = 1, but so is a rectangle with a base of 0.5 and a height of 2 - we can get an area of 1 with a height that's greater than 1 if the base is narrow).

The important point about all this is: continuous probability distributions give us probability densities for single points, not probabilities. Probability for a single point doesn't exist.

This is not a problem for us when we use the normal distribution as a likelihood function - probability densities work just fine as likelihoods. Converting the normal probability density function to a normal likelihood function is again a simple matter of changing notation, and then completely re-thinking what the equation means - that is:

L ( μ ∣ x , σ ) = 1 2 π σ 2 e − 1 2 [ x − μ σ ] 2

The likelihood of any value for the now unknown parameter μ given a known value for x and σ is calculated with the same equation as we used to calculate probability density. The only thing that's different is that we consider the data to be known (because, of course we know the data values, we measured them), and consider μ to be unknown. Considering σ to also be known is not very realistic, but for now we're going to do it anyway as a simplifying assumption - it lets us work with one unknown at a time, which makes the explanations simpler.

Estimating the unknown value of μ given a data value is just like the procedure used with the Poisson distribution above - the likelihood of a range of values for μ can be calculated, and the one that maximizes the likelihood is the best estimate for μ. The app below puts a likelihood function over a data value (selected from a pick list above the curve) and then allows you to select various possible values for μ with a slider - the dot on the curve shows the likelihood of the currently selected value, and you'll see that the maximum likelihood given a single data value is always located at the data value (just like we saw with the Poisson likelihood function).

The likelihood calculation is also shown to the right of the curve as an equation - since 1/sqrt(2πσ2) is a constant it remains the same (0.1) regardless of the value of μ selected. Once a data value is selected from the list it takes the place of x in the equation and remains the same as well, so only the value of μ changes - for a given selected data point the knowns stay the same while the unknown is varied to find the value that maximizes the likelihood. You can hover over the dot to confirm that its likelihood is the one that is being calculated by the equation.

Give it a try with a few different data values - no matter which one you pick the maximum likelihood estimate of μ given a single data value is the data value itself.

It's often convenient to work on a log scale with likelihoods, for...reasons - for the moment take my word for it, you'll see why a little further down. For now though, what we mean by working on a log scale is to take the log of the likelihood function, which converts it to:

log L ( μ ∣ x , σ ) = − 0.5 ln ( 2 π ) − 0.5 ln ( σ 2 ) − 1 2 σ 2 ( x i − μ ) 2

If you drop down the "Choose a likelihood function" menu and select "Log likelihood" you'll see that working on a log scale changes the y-axis scaling but not the x-axis - as such, the shape of the curve changes from bell-shaped to a parabola, but the function is still symmetrical around the data value, and the x-axis location of the maximum is unchanged. Consequently, working on a log scale doesn't change the maximum likelihood estimate for μ.

Finally, since the log of decimals are negative (that is, since 1/10 is also 10-1 the log base 10 of 10-1 is -1), and likelihoods will frequently be decimal numbers, we can make log likelihoods positive values by multiplying by -1. You can see this version of the likelihood function by selecting the "Negative log likelihood" option - this has the effect of flipping the log likelihood vertically around the x-axis, from concave downward to concave upward. The maximum of the likelihood function becomes the minimum of the negative log likelihood, but the location of the best estimate for μ remains at the known data value regardless.

Likelihood with samples of data

This next app illustrates maximum likelihood estimation of the mean for the entire set of 15 data values from the pick list above. We can use exactly the same likelihood function we used for a single data value, but we just need to multiply the likelihoods given each single data value together - that is, the likelihood of μ given the entire data set in the drop-down menu above is:

L ( μ ∣ x i , σ ) = ∏ 1 2 π σ 2 e − 1 2 [ x i − μ σ ] 2

The only change here is that we have a whole set of x values, so we have xi as the known values (to indicate there's more than one), and just like we did with our Poisson likelihoods for two points we are now multiplying our likelihoods together - a capital pi symbol, Π, means "multiply these values together", just like capital sigma, Σ, means "add these values together".

On a log scale multiplication becomes addition, so the log-likelihood is:

log L ( μ ∣ x , σ ) = − 0.5 n ln ( 2 π ) − 0.5 n ln ( σ 2 ) − 1 2 σ 2 ∑ ( x i − μ ) 2

Since we have n likelihood functions, one for each point, the likelihoods are added together n times. Adding a constant n times is the same as multiplying it by n, so both of the terms with constants are multiplied by n now. The squared differences between data values and the estimate for μ are summed.

The negative log-likelihood is this log likelihood multiplied by -1.

The likelihood for the entire data set is the product of these individual likelihoods, and the likelihood function obtained by multiplying all 15 individual likelihood functions together is the graph on the right - the red dot on that graph indicates the likelihood of the current value for μ for the entire data set.

Really, likelihood functions are not probability distributions

The "conversion" of a probability distribution into a likelihood may have seemed less than convincing... if the working end of the thing is exactly the same isn't it still a probability distribution?

One more way to demonstrate that likelihood functions are really something different is to drop unnecessary terms from the likelihood function. For the log likelihood function:

log L ( μ ∣ x , σ ) = − 0.5 n ln ( 2 π ) − 0.5 n ln ( σ 2 ) − 1 2 σ 2 ∑ ( x i − μ ) 2

any additive constant can be dropped (that is, constants that are only added to μ). With additive constants dropped the likelihood function becomes:

log L ( μ ∣ x , σ ) = − 1 2 σ 2 ∑ ( x i − μ ) 2

The effect of dropping these terms is to shift the curve up along the y-axis, but the curvature isn't affected at all. And, if the curvature is the same then the location of the maximum is the same:

The size of the likelihood doesn't matter, so the upper curve is just as usable as a log likelihood function as the one that retains everything from the normal density function. But, a likelihood function that has terms dropped would not in any way be a probability distribution - those constants are needed to make the area under the curve equal 1, and a function with an area under the curve of any other number can't be a probability distribution.

So, while likelihood functions are derived from probability distributions they really aren't probability distributions.

Numerical vs. analytical solutions

In a manner of speaking, what we have been doing by moving a slider until we get a dot to the top of a curve is a form of numerical method of estimating the mean for the data. Numerical methods work by trying out possible values until the correct one is found - our version of this process is very coarse and imprecise compared with the actual numerical analysis methods available (R uses numerical methods for all of its lm() solutions).

But, it's possible to use maximum likelihood to find exact, analytical solutions as well. Analytical solutions are obtained by solving an equation for an unknown, and can be mathematically proven to be the correct solution from the infinite set of possibilities.

Although analytical solutions have the virtue of being correct where numerical methods can only give approximations, numerical approximations can be very good, such that the difference between analytical and numerical solutions are small enough to safely ignore. R uses numerical methods for its lm() function, and they work very well.

But just to show you how it's possible to get analytical formulas for maximum likelihood estimates, let's get the maximum likelihood estimator for μ.

Likelihoods of linear models given the data

The first example of estimating the mean of a sample of data was (relatively) easy to understand, but doesn't really do much useful work for us. We can take the same approach, however, to finding a linear model that has the highest likelihood given the data, and in finding it we will get maximum likelihood estimates for the slope (β) and intercept (α) in the process. From there the world will be our oyster.

Using a straight line, with a slope and an intercept to estimate, seems like it would complicate the likelihood function, but it actually doesn't - the only thing we need to do differently is to replace μ in the likelihood function with the predicted value of the regression model. For example, the log likelihood function looks like this:

log L ( α , β ∣ σ, y i ) = − 0.5 n ln ( 2 π ) − 0.5 n ln ( σ 2 ) − 1 2 σ 2 ∑ ( y i − y ^ ) 2

The unknown parameters we are estimating are on the left side of the vertical line (α,β), and the known data values (yi) and standard deviation of the residuals (σ) are on the right side of the vertical line (remember that the data values are actually known, and we're treating σ as though it's known to keep this a little simpler).

Notice what we're calculating the likelihood of, (yi - ŷ)2 ... if ŷ is the predicted value from a regression line and yi are the data values then yi - ŷ are the residuals for the regression line. We will find values of the slope and intercept that maximize the likelihood by making the squared residuals as small as possible across the entire data set. This is important: we are estimating the values of α and β but they don't even have to appear in the likelihood function directly to do so. If we were to put in α + βx into the function in place of ŷ we would see them, but expressing the function in this way encourages us to think of what we are doing as using a model to predict values for y, and then finding values for α and β that maximize the likelihood of the model given the data.

2. We now have two different models, and the quad model at least appears to fit the data better than the straight line. According to Edwards, 1992:

“Within the framework of a statistical model, a particular set of data supports one statistical hypothesis better than another if the likelihood of the first hypothesis is greater than the likelihood of the second hypothesis”

We can define a measure of the difference in likelihood of two models called support - the degree of support for one model compared to another is calculated as:

Support = log ( L ( H 1 ) L ( H 2 ) ) = log ( L ( H 1 ) ) − log ( L ( H 2 ) )

That is, support is the log of the ratio of the model likelihoods. The log of a ratio is equal to the difference between the log of the numerator and the log of the denominator, so we can calculate the support from our log-likelihoods.

Since we are using the same data set for both models, and the models are nested we can test this with a likelihood ratio test. Nested models are ones in which all the parameters for the simpler model are also found in the more complex model - that is, the simpler model has a subset of the parameters in the more complex model. This is what we have - the linear regression has everything the quadratic model has except for the quadratic term, so the linear regression is nested within the quadratic regression.

The likelihood ratio test uses the negative of two times the support as its test statistic - in other words:

G = -2 (ln(L(H1)) - ln(L(H2)))

The full model, with all of the terms, is H2, and the simpler model that is a nested subset of the first is H1.

This G statistic follows the Chi-square distribution, with degrees of freedom is equal to the difference in model degrees of freedom for the two models - there is only one predictor different between them, so the difference in df is also 1. To test for a difference in support for the linear and quadratic models we compare our G value to a Chi-square distribution with 1 degrees of freedom.

Is the best-supported hypothesis any good at all?

Note that our likelihood ratio test gave us a comparison of two models: simple linear regression and quadratic regression. In this case we found that the quadratic regression fit better than the simple linear regression, but we don't get a test of the linear regression model. As we move into model-based inference we don't want to give up one very nice feature of null hypothesis testing - namely, the null hypothesis. A null hypothesis explicitly hypothesizes a complete lack of effect, and null hypothesis testing uses it as the default that is only rejected if the data gives us reason to. We should still consider a hypothesis of no effect so that we know whether we have any reason to think our predictors have any effect on the response at all.

A model-based approach to testing a null hypothesis is to fit an intercept only model to the data and compare it to the linear regression model. A model that only has an intercept hypothesizes no relationship between y and x at all - or, if you prefer, it hypothesizes that the relationship is a flat line with a slope of 0 and an intercept at ȳ, like the graph on the left.

We can calculate the likelihood of the intercept-only model given the data the same way that we calculated the log likelihood of this model by going back up to the linear regression app and setting the intercept to the mean of the Y data (8.82), and the slope to 0, and selecting log likelihood from the drop-down menu. Since the intercept-only model has a subset of the terms that the regression model has, we can compare them with a likelihood ratio test to see if the regression model has a significantly higher likelihood. Enter the log likelihood of the intercept only model as the simpler model, and the linear regression log likelihood as the more complex model. Since the only difference between them is the slope of the linear regression there is still only 1 DF for the test. You should see that the linear regression fits statistically significantly better than a flat line.

Using R for likelihood ratio testing

Likelihood ratio tests comparing models are simple to do in R, using our familiar lm() model fitting.

2. Fit a linear regression of y as the response against x as the predictor, and store the output in linear.lm. To see the coefficients put the name of the object, linear.lm, in the next row after the model is fitted - you should see they are very similar to what you got with the apps (these are of course better estimates, the apps above are pretty crude, but they should be close). Do this step in chunk linear.regression of your Rmd file.

Likelihood of proportions, and profile likelihood confidence intervals

Proportions are commonly encountered in Biology, and are very simple quantities to calculate. If we had a sample of 20 individuals, of which 15 were heterozygous for a trait we were interested in, we could calculate the proportion that are heterozygous by dividing the number of heterozygotes by the total number observed, or p = 15/20 = 0.75 (don't confuse this for a p-value from a hypothesis test, we are using p to represent the proportion of heterozygotes for this example). Expressing the data as a proportion allows us to compare between samples with different numbers of individuals, and they are very simple, common values to work with.

But, just like a sample mean, the proportion of heterozygotes in a sample is a sample-based estimate of a population parameter. We might like to know, given that we have 15 heterozygotes out of 20 sampled, what range of possible proportion of heterozygotes in the population is likely given the data we have. In other words, we would like to be able to attach a confidence interval to this value of p.

This is a simple enough thing to want, but a surprisingly difficult thing to have. The way we calculate confidence intervals for sample means is to calculate a measure of sampling variation (the standard error), multiply it by a t-value that will capture 95% of the possible sample means from the population, multiply them together to get a measure of uncertainty (t x se), and then add the uncertainty to, and subtract it from, the mean. For a proportion the formulas are a little different, starting with the formula for the variance - with a proportion, p, the variance is:

σ2p = p(1-p)

The standard error is thus:

s . e . p = p ( 1 − p ) n

With a proportion of 0.75 and a sample size of 20 the standard error is s . e . p = 0.75 ( 0.25 ) 20 = 0.097

With proportions we don't need to use the t-distribution, we can use the normal distribution in our confidence interval calculation - since 95% of a normal distribution falls between -1.96 and 1.96, we use 1.96 as the multiplier to calculate the uncertainty. We then add and subtract the uncertainty from p to get our 95% confidence interval:

0.75 ± 1.96 (0.097) = 0.56 to 0.94

Seems to work, right? There are some limitations to this approach though. First, the interval is obtained by adding and subtracting the same number to p, which gives us symmetrical confidence intervals. Given that the we can't get proportions lower than 0 or higher than 1 we should have a greater chance of the proportion falling below 0.75 than above it. We can see the problem even more clearly if we calculate the interval for a proportion closer to 1, like 0.9:

0.9 ± 1.96 0.9 ( 0.1 ) 20 = 0.77   to   1.03

Given that it isn't possible for a proportion to fall above 1 the upper limit is clearly wrong. We're fortunate when the answers we get are transparent nonsense, because we can recognize the error - it is still problematic to have symmetrical confidence intervals with p = 0.75, but the answers are plausible so the problem is less obvious then.

There are various ways to solve this problem that use the same basic approach, but with an adjustment to make the intervals asymmetrical and force them to stay between 0 and 1. None of those are very satisfactory.

A better solution is to use the likelihood function to obtain our confidence intervals. It can be shown that the two values for the parameter whose likelihoods are 1.92 units below the maximum log likelihood are good estimates for the lower and upper 95% confidence interval (this value is 1/2 of the Chi-square critical value of 3.84 with 1 df - again the link between likelihood and the Chi-square distribution is evident).

This sounds complicated, but here's a graphical illustration of how it works:

The upper horizontal line is at the estimate, and has a log likelihood of -11.2467.

The lower line is 1.92 units below the maximum. The two places where the lower horizontal line intersects the log likelihood curve are the values for the lower and upper 95% CI, which are falling around 0.55 and 0.9 on this illustration. You can try this out for yourself below.

The likelihood function for a proportion is simple - it's pn(1-p)K-n, where p is the proportion of heterozygotes, n is the number of heterozygotes, (1-p) is the proportion of not heterozygotes (i.e. the homozygotes), and K - n is the number of not heterozygotes (K is the total individuals, n is the number of heterozygotes, so K - n is the number of homozygotes). The likelihood function for a range of possible values of p is shown in the graph to the right.

1. The app is set to 15 heterozygotes out of 20, which is a proportion of 0.75. The current estimate for the value of p is selected by the slider, which begins at 0.5. It's convenient to think of proportions as the balancing points for count data - the graph on the left shows bars for homozygotes and heterozygotes, and you can think of the x-axis as a teeter-totter. The current value of 0.5 is in the middle of the teeter totter, which makes it unbalanced - the greater number of heterozygotes are weighing down the right side and making the graph tilt. If you move the slider to the right you'll see that you get the graph balanced at p = 0.75, which is also the maximum likelihood estimate according to the graph on the right.

2. With the slider set at 0.75 change the type of likelihood to log likelihood. You'll see that the log likelihood has a much flatter top, but it does have a maximum value, and it's the same place as the peak of the likelihood function.

If you hover over the red dot you'll see that the log likelihood for p = 0.75 is -11.2467. The values that are 1.92 units below that will be at a log likelihood of -11.2467 - 1.92 = -13.1667. If you move your mouse along the likelihood function to the left until you find a value of -13.12544 (the closest my app will let you get to -13.1667) at p = 0.54 (you can see the value for p in a little black text box that pops up below the x-axis) - this is the lower limit for the 95% confidence interval.

If you do the same thing above 0.75, you will find the value of -13.09333 (which is the closest the app gets to -13.1667 on the upper end of the curve) at p = 0.9, which is the upper limit.

So, according to our likelihood function, given our ML estimate of p = 0.75 we can have 95% confidence that the true value falls between 0.54 and 0.9. Because the likelihood function is not symmetrical this method gives us asymmetrical intervals, which makes sense - there is more space below 0.75 than above, so we can be more wrong below than above as well. And, because the likelihood function is bounded at 0 and 1 we can't get ridiculous values as our limits, like values below 0 or above 1.

3. Change the number of heterozygotes to 4, keeping the total at 20, which is a p of 4/20 = 0.2. Set the slider to 0.2, and confirm you're at the maximum likelihood estimate. Then, find the log likelihood at p = 0.2, calculate the value that is 1.92 units below this maximum value, and then find the lower and upper limits. Report these in your Rmd file.

4. Any method of estimating confidence intervals worth its salt should get narrower the more data we collect. This would clearly be true with our standard error-based method, since N is in the denominator of the standard error calculation. It's not as obvious what larger sample sizes would do to the shape of the likelihood functions, so let's see.

Set the total to 100 and the number of heterozygotes to 20 - this also gives us a p of 20/100 = 0.2. If you do your profile likelihood with this larger data set you'll find the log likelihood is much more curved around the estimate, and it gives us a narrower confidence interval as a result. Report the confidence interval for this larger sample size in your Rmd file.

This method of using the likelihood function to find a confidence interval is called profiling the likelihood function, and the interval is a profile likelihood confidence interval.
